================================================================================
ENDPOINT MEDIA — CONTENT CONSOLE (csmm / csmmendpoint)
================================================================================

WHAT THIS PROJECT IS
--------------------------------------------------------------------------------
A Streamlit-based internal tool for a social/content studio ("Endpoint Media").
It stores client brand profiles in SQLite, runs AI-assisted post generation
(CrewAI + Gemini) per client, and tracks manual distribution workflow: captions,
1:1 and 9:16 image prompts, optional overlay JSON, final exported images, QC,
and a separate "publisher" view for someone who posts to Facebook without
needing Meta page access.

Core idea: one place to generate and approve content, hand off assets + copy
to a publisher via a private link (query param + shared key), not auto-post
to Facebook in Phase 1.

KEY FILES
--------------------------------------------------------------------------------
  src/app.py          Main UI, theme CSS, navigation, generation hub, post library,
                      publisher queue view (?view=publisher&key=...)
  src/main.py         Entry: local runs spawn Streamlit on app.py; Cloud runs
                      app in-process (no nested Streamlit server).
  src/database.py     SQLite schema, migrations, posts/clients CRUD, QC/publisher
                      fields, content gap analysis helpers.
  src/seed_clients.py Default clients inserted when DB has zero clients (fresh deploy).
  src/crew.py         CrewAI social media crew wiring.
  src/config/         brand_vault.py (tyre co-brands), vertical_creative.py (per-vertical
                      rules), agents.yaml, tasks.yaml (CrewAI task text + JSON contract).
  static/titan_ad_exporter.html   "Overlay studio" — browser HTML exporter (html2canvas).
  .streamlit/config.toml           Streamlit theme tokens (aligns with app CSS).
  agency.db           Local SQLite file (gitignored). Cloud gets its own ephemeral DB.
  .env.example        Documents env vars; real secrets go in .env (local) or
                      Streamlit Cloud Secrets (deployed).
  scripts/            upsert_* / add_client_* — programmatic client profile updates into agency.db

USER-FACING VIEWS (SIDEBAR NAVIGATION)
================================================================================

The left sidebar is the primary navigation. Structure:

  Endpoint Media          (title — Crimson Pro, large)
  Content studio          (caption — muted)

  [ Dashboard ]           Primary nav — generation + post library for selected client
  [ New client ]          Onboarding form — add/edit client DNA (no generation chrome)
  [ Overlay studio ]      Full-width embed of Titan-style HTML ad exporter

  CLIENTS                 (section heading — uppercase, letter-spaced, muted)
  [ Client name ]         One button per client; active client highlighted (navy).
                          Clicking a client switches active client and returns to Dashboard.

Session state:
  - current_view: Dashboard | New client | Overlay studio
  - em_active_client_id: which client is selected for Dashboard generation/library

SPECIAL URL (NO SIDEBAR FLOW)
--------------------------------------------------------------------------------
  ?view=publisher&key=<PUBLISHER_SHARED_KEY>
  External publisher queue: posts marked QC "Ready for Publisher" for clients
  matching PUBLISHER_QUEUE_CLIENTS. Download ZIP, copy caption, track schedule/posted.

DASHBOARD LAYOUT (MAIN WORK AREA) — "Dashboard" VIEW
================================================================================

When "Dashboard" is selected and at least one client exists:

1) HERO (top)
   - Title: "Endpoint Media — Content console" (class ui-hero)
   - Subtitle explaining: pick client in sidebar, generate posts, copy assets,
     track delivery (class ui-hero-sub)

2) ACTIVE CLIENT STRIP
   - Line: "Active client: <name> · profile #<id>" (class ui-active-client)
   - Left accent bar in bronze (C_ACCENT); border and light surface background

3) CONTENT GAPS (last 30 days)
   - Heading + gap analysis alerts from database (pillars / featured brands vs recent posts)
   - Each alert: info/warning styling; optional "Generate Post to Fill Gap" button

4) GENERATION HUB (contained "card" region)
   - Wrapped in a bordered, shadowed container (em_generation_hub styling)
   - Typically includes:
     - Post format: Standard Feed (1:1) vs Story (9:16)
     - Content pillar, featured brand (vault), creative hook (options vary by vertical:
       automotive tyre vs battery vs firewood)
     - Batch count, mixed-format options, battery product line (battery vertical only)
     - Plan summary panel (em-panel): explains what will run for this client
     - Buttons: "Run generation" (primary navy), "Auto-Pilot (random settings)"

5) POST LIBRARY (below generation)
   - Section header styled as audit block (em-audit-header-*):
     "Post library" / subtitle about asset delivery & tracking
   - Summary table (Streamlit dataframe): post #, created, pillar, brand, format,
     delivery workflow status, QC status
   - Per-post cards (em-card):
     - Suggested caption (read-only text area) + Copy button (clipboard iframe)
     - 1:1 and 9:16 AI image prompts + Copy buttons
     - Optional expander: suggested text overlay JSON
     - Meta line: pillar, brand, format
     - Final images: upload square + vertical, Save, Mark ready for publisher,
       Revert QC to draft
     - Delivery status select: Draft / Sent to Client / Posted

AI GENERATION MODEL — HOW EACH CUSTOMER / VERTICAL "THINKS"
================================================================================

This section explains the rules the app encodes so CrewAI output stays on-brand. The
"model" here is the combination of: CrewAI agents, task prompts (tasks.yaml), injected
vertical rules (vertical_creative.py), optional tyre brand vault (brand_vault.py),
and each client row in SQLite (brand_context, services_list, photography_style, etc.).

PIPELINE (same for all clients)
  1) User selects client + options in Dashboard (format, pillar, featured brand where
     applicable, creative hook, batch).
  2) App builds inputs: company profile + vertical_mode + vertical_creative_rules +
     research_vertical_hint + featured_brand / brand_guidelines / brand_models (tyres only).
  3) CrewAI runs sequential tasks:
     - market_researcher (research_task) — bullets grounded in client + pillar.
     - content_creator (content_creation_task) — ONE JSON object per run:
         Caption,
         Image_Generation_Prompt_1_1,
         Image_Generation_Prompt_9_16,
         Suggested_Text_Overlay { "Heading", "Footer" }.
  4) App saves post; images are produced manually from prompts (Phase 1).

VERTICAL ROUTING (src/config/vertical_creative.py)
  The app picks exactly one vertical per client. That controls which block of rules is
  injected and which creative hooks appear in the UI.

  vertical_mode values:
    firewood   — Miwesu (and similar if name/brief matches).
    battery    — Alberton Battery Mart (name match + battery keywords).
    offroad    — Absolute Offroad (name match) or strong 4x4/offroad keywords in brief.
    automotive — default: tyre retail (e.g. Alberton Tyre Clinic).

--------------------------------------------------------------------------------
AUTOMOTIVE (TYRE RETAIL) — e.g. Alberton Tyre Clinic
--------------------------------------------------------------------------------
How it "thinks":
  - Featured manufacturer (optional): dropdown uses brand_vault.py — slogans, visual rules,
    SA-oriented model hints for grounding copy (no invented tyre SKUs when co-branded).
  - Content pillar routes SETTING: promo/authority → clean studio hero; educational →
    minimal/macro; service → authentic workshop from photography_style.
  - Mechanical realism: wheel balancing shows tyre ON a rim (not loose carcass on balancer);
    alignment shows vehicle on rack or believable hardware — not floating tyre-only rigs.
  - Tyre condition logic: new tyres look new when being fitted; worn only when story needs it.
  - Demographics: if people appear and photography_style does not forbid faces, prompts ask
    for diverse, believable SA representation.
  - Image prompts: dual aspect (1:1 + 9:16), shallow DOF, overlay safe-zones for 1:1.

Hooks (examples): workshop, product hero, did-you-know, promo — tyre-appropriate list in app.py.

--------------------------------------------------------------------------------
BATTERY — Alberton Battery Mart
--------------------------------------------------------------------------------
How it "thinks":
  - NOT tyre retail: image prompts must not centre tyre tread, tyre-only heroes, or tyre
    workshop as the product story.
  - Workshop look when in-store: clean modern SA fit-out (light face brick, floated cement,
    charcoal feature walls) — from BATTERY_WORKSHOP_VISUAL_SPEC in vertical_creative.py.
  - Product lines: Eco Plus / Power Plus are priority for VALUE/showcase/install posts.
    Diagnostics/testing copy should focus on the CUSTOMER'S existing battery and alternator
    health — not “testing a brand-new Eco Plus/Power Plus because it is faulty.”
    New batteries appear in install, product hero, and promo contexts without fault implication.
  - Optional battery line picker in UI for targeted runs.
  - Scene mix: not every ad must be in-store — allowed: premium studio / magazine / gradient
    backdrops, product-on-plinth, campaign-style lighting (still photorealistic, no fake UI).

Hooks: battery-specific list (diagnostics, callout, Start/Stop, fleet, backup power, etc.).

--------------------------------------------------------------------------------
FIREWOOD — Miwesu Fire wood
--------------------------------------------------------------------------------
How it "thinks":
  - No tyre metaphors, no tyre vault co-branding for images/caption.
  - Wood imagery must follow SA hardwood xylology (Geelhaak, Knoppiesdoring, Sekelbos, etc.) —
    ban generic Northern pine/oak/birch look; details in MIWESU_WOOD_VISUAL_SPEC.
  - No phones, tablets, chat UI, or app screens in images — WhatsApp stays in caption CTA only.
  - Product names, bag sizes, MOQs, zones only from brief — no invented prices or suburbs.

Hooks: braai, delivery, product hero, ember/abstract heat, educational macro, etc.

--------------------------------------------------------------------------------
OFFROAD — Absolute Offroad
--------------------------------------------------------------------------------
How it "thinks":
  - Not tyre retail: tyre manufacturer vault and TYRE RETAIL block in tasks.yaml are skipped.
  - South Africa market guardrails: advertise products/brands that are credible SA-available
    per brief; no invented SKUs or international-only lines presented as in-stock.
  - Exact product lock: both image prompts must lead with a line like
    "Exact product to advertise: <one specific product string>" and the caption must repeat
    the same string — no vague "similar to EFS/Tough Dog" style-only wording.
  - Product ads with a user-supplied reference image: prompts repeat strict attachment
    fidelity — study attachment, match geometry/colour/branding, do not mutate the product.
  - Creative hooks are offroad-specific (suspension authority, protection, recovery, platform
    angles, etc.), not generic tyre hooks.

--------------------------------------------------------------------------------
SHARED JSON & QUALITY RULES (tasks.yaml — content_creation_task)
--------------------------------------------------------------------------------
  - Every run outputs valid JSON only (no markdown fences): Caption + two prompts + overlay.
  - Captions end with contact block from brand_context (phones, WhatsApp, site, address rules
    per client), plus one line of 3–8 hashtags.
  - Dual prompts must match aspect (1:1 square vs 9:16 vertical) and usually share the same
    creative idea with recomposed framing.
